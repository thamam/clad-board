# Bot Monitoring Dashboard — System Specification

[Version: 1.0]
[Created: 2026-02-25]

---

## 1. System Overview

A unified, web-based monitoring dashboard for AI agent bots. The dashboard provides real-time visibility into bot health, configuration, and resource usage across all registered bots regardless of their underlying framework.

### Core Design Principles

- **Bot agnosticism** — The dashboard has no direct knowledge of bot internals
- **Sidecar pattern** — A lightweight agent runs alongside each bot, reads its logs, and forwards structured telemetry to the dashboard
- **Interface contract** — Each bot *class* implements a structured log schema once; all instances of that class are automatically monitored
- **Zero bot friction** — The bot process is never modified for monitoring purposes. If data exists in logs, it can be monitored. Only data that doesn't naturally appear in logs requires a schema addition, handled as a one-time per-class effort during registration
- **Monitoring scope only** — This version covers observability only. Command & control (on/off, direct bot interaction) is explicitly out of scope

### Reference Bot Classes

- **OpenClaw** — Primary agent framework (Python-based, multi-model routing)
- **Letta** — Secondary agent framework

## 2. Architecture

```
Bot Process (any class)
  └── emits structured JSON logs
        ↓
Sidecar Container (per bot instance)
  └── tails log file or stdout
  └── parses against log schema
  └── authenticates with registration token
  └── pushes telemetry to Dashboard API
        ↓
Dashboard Backend
  └── receives telemetry
  └── stores and aggregates data
  └── serves Dashboard UI
        ↓
Dashboard UI (dashboard.neuronbox.ai)
  └── secured web interface
  └── unified view across all bots
```

### Deployment

- Hosted on existing AWS EC2 instance alongside bot infrastructure
- New Docker Compose service in existing stack
- New Caddy subdomain: `dashboard.neuronbox.ai`
- DNS record added via GoDaddy (same pattern as `openclaw.neuronbox.ai`)
- Caddy handles HTTPS automatically

## 3. Registration Flow

Registration is a one-time setup per bot instance. The bot itself does not participate.

**Steps:**
1. Operator accesses the dashboard admin panel
2. Creates a new bot entry: name, class (e.g., openclaw), instance label
3. Dashboard generates a unique registration token
4. Operator adds the sidecar container to the bot's Docker Compose stack, passing the token and log path as environment variables
5. Sidecar starts, connects to dashboard using token, and begins forwarding telemetry

**Registration record fields:**
- `bot_id` — UUID, generated by dashboard
- `name` — Human-readable name (e.g., "OpenClaw Production")
- `class` — Bot framework class (e.g., openclaw, letta)
- `token` — Auth token for sidecar-to-dashboard communication
- `registered_at` — Timestamp
- `status` — active / inactive

## 4. Log Schema (Interface Contract)

Each bot class must emit structured JSON log lines. This is the interface contract — implemented once per class.

### Log Line Format

```json
{
  "timestamp": "ISO8601",
  "bot_id": "uuid",
  "event_type": "string",
  "payload": { }
}
```

### Event Types

#### `startup`
Emitted once when the bot starts. Contains all static configuration.

```json
{
  "event_type": "startup",
  "payload": {
    "version": "1.2.0",
    "models": ["gemini-2.5-flash", "claude-sonnet-4-6"],
    "channels": ["telegram", "whatsapp"],
    "skills": ["summarize", "web_search"],
    "tools": [
      { "name": "calculator", "enabled": true },
      { "name": "code_runner", "enabled": false }
    ]
  }
}
```

#### `heartbeat`
Emitted on a regular interval (e.g., every 60 seconds) to confirm the bot is alive.

```json
{
  "event_type": "heartbeat",
  "payload": {
    "uptime_seconds": 3600
  }
}
```

#### `message`
Emitted on each message processed.

```json
{
  "event_type": "message",
  "payload": {
    "direction": "in | out",
    "channel": "telegram",
    "session_id": "optional"
  }
}
```

#### `token_usage`
Emitted after each LLM call.

```json
{
  "event_type": "token_usage",
  "payload": {
    "model": "gemini-2.5-flash",
    "tokens_in": 512,
    "tokens_out": 128
  }
}
```

#### `error`
Emitted on any caught exception or failure.

```json
{
  "event_type": "error",
  "payload": {
    "message": "Connection timeout to Telegram API",
    "severity": "warning | error | critical"
  }
}
```

## 5. Sidecar Specification

The sidecar is a lightweight container added to each bot's Docker Compose stack.

### Responsibilities
- Tail the bot's log file or stdout stream
- Parse JSON log lines matching the schema
- Authenticate to the Dashboard API using the registration token
- Forward parsed events to the Dashboard API
- Handle connection failures gracefully (buffer and retry)

### Configuration (environment variables)
```
DASHBOARD_URL=https://dashboard.neuronbox.ai
REGISTRATION_TOKEN=<token from dashboard>
LOG_PATH=/logs/bot.log   # or "stdout" for container stdout
```

### Docker Compose snippet
```yaml
sidecar:
  image: bot-dashboard-sidecar:latest
  environment:
    - DASHBOARD_URL=https://dashboard.neuronbox.ai
    - REGISTRATION_TOKEN=${BOT_TOKEN}
    - LOG_PATH=/logs/bot.log
  volumes:
    - ./logs:/logs:ro
  restart: unless-stopped
```

### Tech Stack Recommendation
- **Language:** Python or Go — whichever produces a smaller, more dependency-free container
- **Log tailing:** `watchfiles` (Python) or native Go file watching
- **HTTP client:** Standard library only — keep dependencies minimal

## 6. Dashboard Backend

### Responsibilities
- Receive telemetry from sidecars (REST API)
- Store and aggregate data
- Compute token usage segments (all-time, last 24h, month-to-date)
- Serve the frontend
- Manage bot registration

### API Endpoints (internal, sidecar-facing)
```
POST /api/ingest          — receive telemetry events (auth: registration token)
POST /api/register        — create new bot registration (auth: admin)
GET  /api/bots            — list all registered bots (auth: session)
GET  /api/bots/:id        — get bot detail + metrics (auth: session)
```

### Data Storage
- **Primary store:** SQLite for MVP (simple, zero-ops, sufficient for this scale)
- **Schema:** events table (raw), bots table (registry), aggregates table (pre-computed segments)
- Upgrade path to Postgres if needed — keep queries standard SQL

### Token Aggregation Logic
Pre-compute and cache on ingestion:
- `all_time` — sum from first event
- `last_24h` — rolling window, recomputed periodically
- `month_to_date` — sum from first day of current calendar month

### Tech Stack Recommendation
- **Language:** Python with FastAPI
- **ORM:** SQLModel
- **Auth:** HTTP Basic Auth for session (MVP), upgrade later

## 7. Dashboard Frontend

### Pages

**Overview (default view)**
- Grid of all registered bots
- Per bot card: name, class, status indicator, last heartbeat, message count, top-level error count
- Status: `online` (heartbeat within last 2 min) / `idle` (heartbeat within last 10 min) / `offline`

**Bot Detail view**
- Identity: name, class, version, uptime
- Configuration: models, channels, skills, tools (with enabled/disabled badge)
- Activity: messages in/out, last message timestamp, active sessions
- Token usage table per model:
  - Columns: Model | Tokens In | Tokens Out | (segments: All Time / Last 24h / Month-to-Date)
- Errors: count, last error message and timestamp

### Tech Stack Recommendation
- **Framework:** React
- **Styling:** Tailwind CSS
- **Data fetching:** React Query (polling, not websockets — MVP)
- **Charts:** Recharts if any visualization needed (MVP: tables only)

---

## 8. Authentication

- HTTP Basic Auth (username + password)
- Configured via environment variable at deploy time
- Caddy can enforce this at the proxy layer if preferred (zero-code auth)
- Upgrade to session-based or OAuth when C&C scope is added

---

## 9. Out of Scope (This Version)

- Command & control (start/stop bots, send messages)
- HTTP endpoints on the bot side
- Real-time websocket streaming (polling sufficient for MVP)
- Multi-user roles or permissions
- Alerting or notifications
- Historical charts or time-series visualization
